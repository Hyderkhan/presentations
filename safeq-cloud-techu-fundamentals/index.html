Deployment options · Choose your control level

Three ways to deploy SAFEQ Cloud.

A spectrum, based on how much data control and operational ownership you need — from standard multi-tenant cloud to fully managed on-premise.

Standard cloud

Multi-Tenant

Shared SAFEQ Cloud with full automation, three-zone HA, and bi-weekly updates. The default for most customers.

  • Per-device licensing
  • Y Soft–operated DCs worldwide
  • Standard SLA
Dedicated

Reserved

Dedicated infrastructure in a dual-zone setup, in the Y Soft–operated DC of the customer's choice. Same platform, isolated environment, premium SLA.

  • Dedicated resources
  • Customer-chosen DC (AWS or T Cloud)
  • Premium SLA included
Maximum control

Managed On-Prem

SAFEQ Cloud deployed into customer-controlled infrastructure. Print and scan data never leaves the network.

  • Customer-operated infrastructure
  • Print & scan data stay on-prem
  • Customer secures the environment

Technology · Container architecture

How we scale: containerization.

SAFEQ Cloud runs inside containers — think of each as a standardized shipping box for an application, with everything it needs packed inside. It's the same architecture Google built to run at planetary scale, and it's how SAFEQ Cloud runs 120K devices across the platform with no degradation.

Traditional on-premise
App on bare server

Apps installed directly on servers — or in costly virtual machines that ship a full OS each time. Scaling means provisioning more hardware. Updates mean downtime windows.

SAFEQ Cloud
App in a container

A standardized shipping box for the application — code, libraries, config. Multiple containers safely share the same OS without VM overhead. (Database stays on managed RDS.)

01 · Auto-scale
Massive · Horizontal
Spin up more containers in seconds as demand grows. No VM provisioning, no infrastructure delay.
02 · Load balancing
Always healthiest
Traffic auto-routes to the healthiest zone and least-loaded container. No hot spots, no manual tuning.
03 · Self-healing
Detect & replace
A failing container is detected and replaced automatically. Updates roll out incrementally with instant rollback.
