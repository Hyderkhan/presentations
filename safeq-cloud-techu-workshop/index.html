Technology · Container architecture

How we scale: containerization.

SAFEQ Cloud runs inside containers — think of each as a standardized shipping box for an application, with everything it needs packed inside. It's the same architecture Google built to run at planetary scale, and it's how SAFEQ Cloud runs 120K devices across the platform with no degradation.

Traditional on-premise
App on bare server

Apps installed directly on servers — or in costly virtual machines that ship a full OS each time. Scaling means provisioning more hardware. Updates mean downtime windows.

SAFEQ Cloud
App in a container

A standardized shipping box for the application — code, libraries, config. Multiple containers safely share the same OS without VM overhead. (Database stays on managed RDS.)

01 · Auto-scale
Massive · Horizontal
Spin up more containers in seconds as demand grows. No VM provisioning, no infrastructure delay.
02 · Load balancing
Always healthiest
Traffic auto-routes to the healthiest zone and least-loaded container. No hot spots, no manual tuning.
03 · Self-healing
Detect & replace
A failing container is detected and replaced automatically. Updates roll out incrementally with instant rollback.

Section 02 · Storage

Storage
Options.

Where the print job lives — on the workstation, in the cloud, or somewhere in between. Pick the trade-off you want.

04 · SectionStorage options
Storage modes · the menu

Three ways to store a print job.

SAFEQ Cloud lets the admin pick where the job sits. Each mode trades cloud egress, sovereignty, and bandwidth differently.

01 · Local

Local Storage

Job stays on the workstation. Only metadata reaches the cloud. Direct print to the device.

  • Smallest cloud footprint
  • Workstation must be online to release
  • Best fit: data-sovereignty critical
02 · Cloud

Cloud Storage

Job and metadata both ride to SAFEQ Cloud. Pull-print from any device, any time.

  • True roaming pull-print
  • Workstation can sleep / shut down
  • Best fit: standard cloud-first office
03 · Hybrid

Hybrid Storage

Direct print local, secure pull-print to cloud. Per-driver decision based on the workflow.

  • Best of both worlds
  • One client, two destinations
  • Best fit: mixed user populations
05 · StorageThree storage modes · trade-offs
Storage mode 01

Local Storage.

The job stays on the workstation. Only the metadata — what was printed, by whom, where — lands in SAFEQ Cloud.

  • Job lives on the PC. Print spool remains local, never leaves the network.
  • Metadata to cloud. Job ID, owner, page count, cost, target device — that's it.
  • Direct print path. PC ➜ MFP, with SAFEQ Cloud auditing the transaction.
  • Bandwidth-cheap. No job egress; ideal for branches with constrained links.
06 · StorageLocal storage · Job stays on the PC
Storage mode 02

Cloud Storage.

Both job and metadata ride to SAFEQ Cloud. Pull-print from any registered device, regardless of workstation state.

  • Job rides to the cloud. Encrypted in transit and at rest in the SAFEQ Cloud region you chose.
  • True roaming pull-print. Release at any printer the user is entitled to.
  • Workstation can sleep. No need for the originating PC to stay awake.
  • Cleanest user experience. The default for cloud-first organisations.
07 · StorageCloud storage · Job + metadata in cloud
Storage mode 03

Hybrid Storage.

One client, two destinations. Direct Print stays local for fast walk-up. Secure Print rides to the cloud for pull-print from anywhere.

Secure Print

Job to cloud, release at any registered device. The cloud-first default.

Direct Print

Job stays local, prints to the named device. Only metadata leaves the network.

08 · StorageHybrid · Secure to cloud, Direct local
Storage mode 04

Print using a Gateway.

The gateway holds the job on-prem. Workstations don't need to stay awake — the gateway plays the role of cloud, locally.

  • Gateway acts as the spool. Holds the job locally, fully managed from the cloud.
  • Print job metadata only leaves the network — sovereignty preserved.
  • Workstations can sleep. The gateway is always available to release.
  • Single managed appliance. Software VM or Y Soft OMNI Bridge hardware.

12 · HybridSoftware vs Hardware appliance
Scenario I · Data sovereignty

Print job content never leaves the network.

  • Gateway fully managed from the cloud. Patching, telemetry, lifecycle — Y Soft.
  • Print job metadata only traverses the firewall.
  • Minimal IT workload. No on-prem print server to patch, monitor, or back up.
  • Sovereignty preserved. Content lives where the regulator wants it.
13 · HybridScenario I · Data sovereignty
Scenario II · Remote branches

Branch offices on thin pipes — without losing print.

  • Gateway per branch. Job stays inside the branch LAN; the link doesn't carry print.
  • OCR processing in cloud. Heavy lift moves to SAFEQ Cloud, results back over thin metadata.
  • Zero-trust regular office coexists with constrained remote sites in the same tenant.
  • Resilient by design. Branch keeps printing if the WAN drops.
14 · HybridScenario II · Remote branches
Scenario III · Legacy systems integration

AS/400 and friends — brought into zero-trust.

  • Legacy systems on-premise. Mainframes, AS/400, ERP modules without modern auth.
  • No end-point authentication, no encryption — or only LPR with no IPP support.
  • Gateway as the bridge. Accepts legacy traffic locally, presents zero-trust upstream.
  • Zero-trust badge release at the device, even when the source can't speak it.
15 · HybridScenario III · Legacy systems
Scenario IV · Legacy devices

Older printers & MFPs — no rip-and-replace.

  • No capability for secure cloud authentication on the device itself.
  • No capability to "download" print jobs from the cloud directly.
  • Applies to printers and older MFP APIs — HP OXPD and similar legacy device APIs.
  • Note: HP WorkPath already supports zero-trust natively on newer devices.
16 · HybridScenario IV · Legacy devices
Scenario V · Direct mobile printing

AirPrint & mobile workflows, governed by SAFEQ Cloud.

  • AirPrint / IPP everywhere — the gateway advertises devices to mobile clients.
  • Secure Print path for sensitive jobs — release at any device after auth.
  • Direct Print path for walk-up convenience — queue to the named device.
  • Same audit trail as wired clients. The cloud sees every job, regardless of source.
